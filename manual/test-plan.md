# Test Plan — Automation Test Store

## 1. Цель тестирования
Проверить корректность ключевых пользовательских сценариев интернет‑магазина Automation Test Store, включая регистрацию, авторизацию, поиск товаров, работу корзины, оформление заказа и функциональность личного кабинета.  
Цель — убедиться, что пользователь может успешно совершить покупку как гость и как зарегистрированный пользователь.

---

## 2. Объект тестирования
Веб‑приложение Automation Test Store:  
https://automationteststore.com/

Основные модули:
- Регистрация и аутентификация
- Каталог товаров и поиск
- Карточка товара
- Корзина
- Оформление заказа
- Личный кабинет
- Страница счёта (invoice)

---

## 3. Области тестирования

### 3.1. Регистрация и авторизация
- Регистрация нового пользователя
- Авторизация с валидными/невалидными данными
- Восстановление пароля
- Проверка валидаций форм

### 3.2. Каталог и поиск
- Поиск существующих и несуществующих товаров
- Переходы по категориям
- Фильтрация (если доступна)
- Просмотр карточки товара

### 3.3. Корзина
- Добавление товара
- Удаление товара
- Изменение количества
- Пересчёт итоговой суммы
- Добавление нескольких товаров

### 3.4. Оформление заказа
- Оформление заказа гостем
- Оформление заказа зарегистрированным пользователем
- Проверка обязательных полей
- Ошибки валидации (email, телефон и т.д.)

### 3.5. Личный кабинет
- Просмотр истории заказов
- Просмотр деталей заказа
- Изменение профиля
- Управление адресами

### 3.6. Сквозные сценарии
- Покупка товара гостем
- Покупка товара зарегистрированным пользователем

---

## 4. Приоритеты

| Приоритет| Область |
|----------|---------|--------------------------------
| Высокий  | Авторизация, корзина, оформление заказа |
| Средний  | Поиск, каталог, личный кабинет          |
| Низкий   | UI‑несоответствия, необязательные поля  |

---

## 5. Подходы к тестированию
- Функциональное тестирование
- Позитивные и негативные сценарии
- Техники тест‑дизайна:
  - эквивалентные классы
  - граничные значения
  - сценарное тестирование
- Ручное тестирование + автоматизация критических сценариев
- Использование режима инкогнито для изоляции сессий

---

## 6. Риски и допущения

### Риски
- Отсутствие публичного API усложняет подготовку данных
- Возможная нестабильность внутреннего API
- Зависимость от cookies и состояния сессии
- Возможные баги в оформлении заказа

### Допущения
- Приложение доступно и работает стабильно
- Данные пользователей не удаляются автоматически
- Внутренние API‑запросы не меняются в процессе выполнения задания

---

## 7. Нефункциональные проверки
Минимально:
- Время загрузки страниц
- Корректность отображения на разных разрешениях
- Базовая проверка адаптивности

---

## 8. Что не тестируется
- Производительность под нагрузкой
- Интеграции с реальными платёжными системами
- Email‑уведомления
- Мобильные приложения (если есть)
- Админ‑панель (если есть)

---

## 9. Окружение
- ОС: Windows
- Браузеры: Microsoft Edge
- Режим: инкогнито
- Очистка cookies перед тестами
